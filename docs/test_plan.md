# PView Alert System - End-to-End Test Plan

This document provides a comprehensive, step-by-step test plan to verify the entire lifecycle of the PView Alert System on a clean database deployment. Follow these steps on your development server (`http://3.108.182.22:8081/pview_alerts/`) to ensure all features function as expected.

---

## Phase 1: Initial Setup & Access Control

**Objective:** Verify that the system allows logging in and correctly enforces dynamic roles and user creation.

### Step 1.1: System Login
1. Open a browser and navigate to `http://3.108.182.22:8081/pview_alerts/`
2. Log in using the default credentials generated by your `setup_default.php` script (e.g., Username: `admin`, Password: `admin`).
3. **Expected Result:** Successful login, redirect to the Dashboard.

### Step 2.2: Create Dynamic Roles
Navigate to **Roles** in the sidebar. We will create two distinct roles.
1. Click **Add Role**. 
   - **Name:** `Global Admin`
   - **Admin Scope:** Toggle ON (This user will see all tickets globally).
   - **Permissions:** Grant `View`, `Add`, `Edit`, `Delete` across all modules in the Module Control Panel.
   - Click **Save**.
2. Click **Add Role** again.
   - **Name:** `L1 Network Support`
   - **Admin Scope:** Toggle OFF (This user will only see tickets assigned to them or their pool).
   - **Permissions:** Grant `View` for Dashboard, Projects, Flows, Tickets. Grant `Add`, `Edit` for Tickets. Do not grant Delete permissions.
   - Click **Save**.
3. **Expected Result:** Both roles appear in the list view.

### Step 2.3: Create Test Users
Navigate to **Users** in the sidebar.
1. Click **Add User**.
   - **Name:** `John Doe (L1)`
   - **Email/Username:** `john_l1`
   - **Password:** `P@ssword123`
   - **Role:** Select `L1 Network Support`.
   - Click **Save**.
2. Click **Add User**.
   - **Name:** `Sarah Admin`
   - **Email/Username:** `sarah_admin`
   - **Password:** `P@ssword123`
   - **Role:** Select `Global Admin`.
   - Click **Save**.
3. **Expected Result:** Users are successfully created.

## Phase 2: Workflow & System Configuration

**Objective:** Verify that Projects, Flows, States, and Alerts can be correctly configured.

### Step 2.1: Create a Project
Navigate to **Projects** -> **Add Project**.
1. **Name:** `IT Infrastructure`
2. **Description:** `Monitoring core networking and server infrastructure.`
3. Click **Save**.
4. **Expected Result:** Project is created and listed.

### Step 2.2: Create a Flow
Navigate to **Flows** -> **Add Flow**.
1. **Project:** Select `IT Infrastructure`
2. **Name:** `Server Outage Incident`
3. Click **Save**.
4. **Expected Result:** Flow is created.

### Step 2.3: Define States & SLA Timers (TAT)
From the Flows list, click **States** for `Server Outage Incident`. We will create a 3-step lifecycle.
1. **Create State 1 (Initial):**
   - **Name:** `New Alert`
   - **Is Initial State:** Checked
   - **Is Final State:** Unchecked
   - **Parent State:** (None)
   - **L1 Users:** Select `John Doe (L1)`
   - **L1 TAT (Minutes):** `5`
   - Click **Save**.
2. **Create State 2 (Process):**
   - **Name:** `Investigating`
   - **Is Initial State:** Unchecked
   - **Is Final State:** Unchecked
   - **Parent State:** Select `New Alert`
   - **L1 Users:** Select `John Doe (L1)`, Select `Sarah Admin`
   - **L1 TAT (Minutes):** `15`
   - Click **Save**.
3. **Create State 3 (Final):**
   - **Name:** `Resolved`
   - **Is Initial State:** Unchecked
   - **Is Final State:** Checked
   - **Parent State:** Select `Investigating`
   - Click **Save**.
4. **Expected Result:** The visual flow tree (vis-network diagram) correctly draws `New Alert --> Investigating --> Resolved`.

### Step 2.4: Create an Alert Definition
Navigate to **Alerts** -> **Add Alert**.
1. **Project:** `IT Infrastructure`
2. **Flow:** `Server Outage Incident`
3. **Alert Name:** `High CPU Usage`
4. **Severity:** `Critical`
5. **Threshold:** Value `95`, Unit `%`
6. Click **Save**.
7. **Expected Result:** Alert definition is saved successfully.

## Phase 3: Ticket Execution & Verification

**Objective:** Verify ticket creation, isolation rules, action tabs, and state progression.

### Step 3.1: Raise a Ticket Manually
While still logged in as `admin`:
1. Navigate to **Tickets** -> **Raise Ticket**.
2. **Project:** `IT Infrastructure`
3. **Flow:** `Server Outage Incident`
4. **Title:** `Server DB-01 Unresponsive`
5. **Description:** `Ping failing to 192.168.1.100`
6. **Severity:** `Critical`
7. **Priority:** `Urgent`
8. Click **Raise Ticket**.
9. **Expected Result:** Ticket is created, you are taken to the list. The status should be `Open`. Note down the **Alarm ID**.

### Step 3.2: Verify Role Isolation (Admin Scope testing)
1. **Log out** of the `admin` account.
2. **Log in** as `john_l1` (Password: `P@ssword123`).
3. Navigate to **Tickets**.
4. **Expected Result:** John should see the `Server DB-01 Unresponsive` ticket because he is an assigned L1 user for the `New Alert` state.
5. Click on the ticket.
6. Verify the visual timeline: `New Alert` is highlighted as Current. The TAT timer should show a live countdown starting from 5 minutes.

### Step 3.3: Take Actions on the Ticket
As `john_l1`, from the ticket detail page:
1. **Action Tab: Comment**
   - Type: `@sarah_admin Please check this outage.`
   - Click Add Comment.
   - **Expected Result:** Comment appears in the Activity Timeline.
2. **Action Tab: Assign**
   - Select `John Doe (L1)` from the dropdown. 
   - Click Assign.
   - **Expected Result:** Assignee is updated, action recorded in timeline.
3. **Action Tab: Attach**
   - Upload a small text file (`ping_results.txt`).
   - **Expected Result:** File uploads successfully and is visible in the timeline.

### Step 3.4: State Progression
1. **Action Tab: Move State**
   - Target State: `Investigating`
   - Click Move State.
   - **Expected Result:** Ticket state updates. The visual diagram updates to show `Investigating` as Current, and `New Alert` as Completed. The TAT timer resets to 15 minutes.

## Phase 4: Final Validation

**Objective:** Verify Dashboard accuracy, bulk features, and terminal states.

### Step 4.1: The Terminal State
1. From the ticket detail page (still as `john_l1`), click the **Resolve** button in the Action tab, or move the state to the final `Resolved` state.
2. **Expected Result:** The ticket status changes to `Resolved`. The action UI (comment, assign, move) becomes locked (read-only). The TAT timer displays "TAT stopped - ticket resolved".

### Step 4.2: Bulk Actions
1. Navigate to the **Tickets** list view.
2. Check the bulk checkbox next to your ticket.
3. Click the **Close Selected** button from the bulk toolbar.
4. **Expected Result:** The ticket status updates to `Closed`.

### Step 4.3: Dashboard Verification
1. Navigate to the **Dashboard**.
2. **Expected Result:**
   - Active Projects should read `1`
   - Active Flows should read `1`
   - Open Tickets should read `0`
   - Resolved / Closed numbers should accurately reflect the ticket you just closed.

### Step 4.4: Auto-Escalation Check (Optional/Advanced)
To test the backend SLA script:
1. Raise a second ticket. Leave it in the `New Alert` state.
2. Wait 6 minutes (so the 5-minute L1 TAT expires).
3. If cron is running on the server, the `php spark tat_monitor run` job should trigger.
4. Check the ticket activity timeline: An automated system entry should read that the ticket breached L1 and escalated.
